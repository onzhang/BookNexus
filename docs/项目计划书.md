## 一、项目概述

### 1.1 项目背景

开发一套面向图书馆的在线图书管理系统，采用**生产级技术架构**设计。虽为个人项目，但技术选型对标企业真实环境，涵盖缓存、消息队列、全文搜索、限流熔断、分布式定时任务等主流中间件。

### 1.2 项目目标

- 实现管理员端全部管理功能（用户、书籍、书架、类别、借阅、通知、公告）
- 实现用户端自助服务功能（浏览、借阅、收藏、留言）
- 构建**高并发、高可用**的后端架构
- 集成 Redis / RabbitMQ / Elasticsearch / Sentinel / XXL-Job 等中间件
- 提供清晰的 RESTful API 供前端调用

### 1.3 系统角色

| 角色 | 描述 |
|------|------|
| **管理员** | 拥有系统全部管理权限，可管理用户、书籍、书架、类别、借阅、通知、公告、日志 |
| **普通用户** | 可浏览书籍、借阅书籍、收藏书籍、查看历史、留言、接收通知 |

**权限模型**：角色枚举判断（非完整 RBAC）
- `user` 表增加 `role` 字段：`ADMIN` / `USER`
- 拦截器校验 JWT 后从 token 提取角色
- `/api/v1/admin/**` 仅允许 `ADMIN` 角色
- `/api/v1/user/**` 允许 `ADMIN` + `USER`

## 三、系统架构设计

### 3.1 整体架构分层

```
┌─────────────────────────────────────────────────┐
│                 接入层 (Controller)               │
│     Sentinel 限流 → 登录拦截(JWT) → 参数校验      │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│                 业务层 (Service)                  │
│     Redis 缓存 → MQ 异步通知 → XXL-Job 定时任务  │
└──────┬──────────────┬──────────────┬────────────┘
       │              │              │
┌──────▼──────┐ ┌─────▼──────┐ ┌───▼──────────┐
│  Redis      │ │  RabbitMQ  │ │  Elasticsearch│
│  (缓存/锁)  │ │  (队列)    │ │  (搜索)      │
└─────────────┘ └────────────┘ └──────────────┘
       │              │
┌──────▼──────────────▼──────────────────────────┐
│              数据访问层 (Mapper/DAO)             │
│        MyBatis-Plus → Druid → MySQL             │
└─────────────────────────────────────────────────┘
```

### 3.2 Redis 缓存设计

| 缓存键模式 | 类型 | 用途 | TTL |
|-----------|------|------|-----|
| `books:hot:{id}` | Value | 热门书籍详情缓存 | 10min |
| `books:list:page:{page}:size:{size}` | Value | 书籍列表缓存 | 5min |
| `categories:tree` | Value | 类别树形缓存 | 30min |
| `user:session:{token}` | Value | JWT Session 存储 | 7天 |
| `borrow:lock:{bookId}` | Redisson Lock | 借阅分布式锁 | - |
| `books:hot:ranking` | SortedSet | 热门书籍排行 | 1h |
| `rate:limit:{ip}:{api}` | Value(递增) | 接口限流计数 | 1s |

**缓存策略**：Caffeine 做一级本地缓存（短 TTL），Redis 做二级分布式缓存（长 TTL），两级缓存穿透回源 DB。

### 3.3 RabbitMQ 事件设计

| 交换机 | 路由键 | 队列 | 用途 |
|--------|--------|------|------|
| `book.exchange` | `book.borrow` | `book.borrow.queue` | 借阅申请异步处理 |
| `book.exchange` | `book.return` | `book.return.queue` | 归还事件，触发订阅通知 |
| `book.exchange` | `book.subscribe` | `book.subscribe.queue` | 订阅通知推送 |
| `log.exchange` | `log.operation` | `log.operation.queue` | 操作日志异步写入 |
| `notice.exchange` | `notice.overdue` | `notice.overdue.queue` | 逾期提醒通知 |
| `dlx.exchange` | `#` | `dlx.queue` | 死信队列（兜底） |

> 所有业务队列均配置死信交换机 `dlx.exchange`，重试耗尽后消息自动转入死信队列。

### 3.4 定时任务（XXL-Job）

| 任务 | 调度频率 | 说明 |
|------|----------|------|
| 借阅逾期检查 | 每天 8:00 | 检查超期未还书籍，推送催还通知 |
| 热门书籍排行 | 每小时 | 从 Redis SortedSet 同步到 DB |
| 订阅通知推送 | 实时（MQ） + 兜底每 10min | 图书归还时推送订阅用户 |
| 缓存预热 | 每日凌晨 | 预加载热门书籍到缓存 |
| 数据统计 | 每日凌晨 | 昨日借阅统计报表 |

### 3.5 搜索架构（Elasticsearch）

```
MySQL ──(同步)──→ Elasticsearch
                  │
                  ├── 书籍标题/作者/ISBN 全文搜索
                  ├── 多字段组合查询
                  ├── 拼音搜索 (pinyin analyzer)
                  └── 搜索建议 (completion suggester)
```

**同步方案**：使用 MQ + 定时任务双重保证：数据变更 → 发 MQ → ES 写入程序消费；每日定时全量同步兜底。

**搜索演进路径**：M3 MySQL FULLTEXT → M5 ES 无缝切换

### 3.6 请求追踪（requestId）

- **方案**：UUID
- **实现方式**：自定义 `HandlerInterceptor`，在 `preHandle` 中生成 UUID
- **存储方式**：`RequestContextHolder`（ThreadLocal）传递
- **自动注入**：`Result` 的 `requestId` 字段在构造时自动填充
- **日志关联**：MDC 中放入 `requestId`，所有日志自动携带 `[%X{requestId}]`

理由：UUID 零依赖、零配置，单机场景足够；Snowflake 对单体系统是过度设计。

## 五、数据库设计概要

### 5.1 核心实体关系

```
用户(User) ──1:N── 借阅记录(BorrowRecord) ──N:1── 书籍(Book)
书籍(Book) ──N:1── 类别(Category)
书籍(Book) ──N:1── 书架(Bookshelf)
用户(User) ──1:N── 收藏(Favorite) ──N:1── 书籍(Book)
用户(User) ──1:N── 留言/建议(Message)
用户(User) ──1:N── 订阅(Subscription) ──N:1── 书籍(Book)
管理员 ──1:N── 公告(Announcement)
管理员 ──1:N── 通知(Notification)
系统 ──1:N── 操作日志(OperationLog)
```

### 5.2 数据表清单（预计 14 张表）

| 序号 | 表名 | 说明 | 优先级 |
|------|------|------|--------|
| 1 | `user` | 用户表（含角色字段区分管理员/普通用户） | P0 |
| 2 | `book` | 书籍表 | P0 |
| 3 | `category` | 类别表 | P0 |
| 4 | `bookshelf` | 书架表 | P1 |
| 5 | `borrow_record` | 借阅记录表 | P0 |
| 6 | `favorite` | 用户收藏表 | P2 |
| 7 | `subscription` | 图书订阅表 | P3 |
| 8 | `notification` | 通知表 | P2 |
| 9 | `message` | 读者留言/建议表 | P3 |
| 10 | `announcement` | 公告表 | P2 |
| 11 | `operation_log` | 操作日志表（MQ 批量写入） | P2 |
| 12 | `book_category_rel` | 书籍-类别关联表（支持多类别） | P1 |
| 13 | `xxl_job_*` | XXL-Job 调度表 | P4 |

### 5.3 审计字段规范

每张表必须包含以下审计字段：

```sql
id          BIGINT       主键（自增）
created_at  DATETIME     创建时间（默认 CURRENT_TIMESTAMP）
updated_at  DATETIME     更新时间（默认 CURRENT_TIMESTAMP ON UPDATE）
is_deleted  TINYINT      逻辑删除标记（0=正常，1=已删除）
```

### 5.4 索引设计要点

- 书籍表：`title` 普通索引，`isbn` 唯一索引，`category_id` / `bookshelf_id` 外键索引
- 借阅记录表：`user_id` + `status` 联合索引，`book_id` 索引，`borrow_date` 索引
- 操作日志表：`operator` + `create_time` 联合索引（MQ 批量写入时注意）
- 收藏/订阅：`user_id` + `book_id` 唯一联合索引

## 八、借阅业务规则

| 参数 | 值 |
|------|-----|
| 每人最大借阅数 | **5 本** |
| 单次借阅期限 | **30 天** |
| 逾期罚款 | **0.1 元/天** |
| 续借次数 | **1 次** |
| 续借延长期限 | **15 天** |
| 续借条件 | 到期前 7 天内可申请，无逾期记录 |

**借阅状态机：**

```
PENDING ──→ APPROVED ──→ BORROWED ──→ RENEWED ──→ RETURNED
    │                                            ↑
    └──→ REJECTED                                未续借直接归还
```

- `PENDING`: 用户提交借阅申请
- `APPROVED` / `REJECTED`: 管理员审批（拒绝需填写原因）
- `BORROWED`: 已确认借出，开始计时
- `RENEWED`: 续借成功，到期日 +15 天
- `RETURNED`: 已归还，借阅流程结束

## 十四、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 中间件版本兼容性 | 项目无法启动 | 统一使用 docker compose 固定版本测试 |
| ES 同步延迟 | 搜索结果不准确 | MQ 保障实时 + 定时任务兜底 |
| 消息丢失 / 未消费 | 通知遗漏 | 死信队列 + 重试机制 + 监控告警 |
| 缓存与 DB 不一致 | 脏数据 | 先删缓存再写 DB + 延迟双删策略 |
| 分布式锁死锁 | 借阅功能瘫痪 | Redisson 自带看门狗机制 + 超时释放 |
| 内存溢出 | 服务崩溃 | 限制 ES 查询深度 + Caffeine 缓存上限 |
| 时间不足 | 延期交付 | P0/P1 优先，P4 功能可裁剪 |
